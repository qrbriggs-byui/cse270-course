CSE 270: Classroom Notes M10: Verification and Validation

Notes for Classroom

Overview

This week is a follow up to Lifecycles, but dealing with the Requirements, Design and Implementation phases specifically, focusing on Functional and Non-functional attributes of Testing characteristics in the process of Verifying and Validating Requirements. Students will implement a test that will address functional and non-functional requirements and design elements. Students will distinguish between validation and verification and learn when to use each. Students will write a System Acceptance Plan and then test it using QA Checklist.

Objectives

By the end of the week, each student will be able to:

  • Differentiate Functional and Non-Functional Testing criteria from Requirements, Design and Implementation
  • Present a customer report
  • Create a Bug Report form
  • Complete bug reports for a product

Announcement

Welcome to Week 11: Verification Validation
This week we will be working on bug reports. After verifying requirements through testing, defects are sure to abound. A bug needs to be documented and recorded properly. This is very important.
Some of the hard bugs to fix are very general. For example, a user reports, “The program stopped working.” This is not a useful description of the bug. Or another one, “The button doesn’t work.” Â There is much more information that needs to be known before it is even close to being solved. For example:

  • What was the user doing?
  • What was running?
  • What feature was being used?
  • What time of day? (For time-sensitive programs)
  • What version of the software is being used?
  • What data is being processed?

All this information is needed and should be recorded in a bug report.
This week you will be creating a bug report template, improving on it, and then using it to write five actual bug reports.
You say, why are we learning to write bug reports? The answer, to learn what to give back to the project’s developer when your test team finds a bug. Completely unprofessional to tell your project developers, “It didn’t pass because it stopped working.” They will want to know the above information to know where to start looking.
There are usually two different types of attributes to testing the features of a system. The first is the Functional requirements. These are requirement-based. For example, the computer must connect to the server to get user information, or a button of a certain size, shape, color, and location is specified. The second is non-functional. These are requirements of the system that are a bit harder to test, such a performance, security, legal, portability, reliability, localization, and usability. Typically non-functional requirements are those requirements outside the scope of the product but affect the product.
Verification is testing to make sure that the Functional Requirements are correct. Does the product do what we said it would do? Validation is testing to ensure that it what the customer wants, needs, can use, and will work for all situations or more of the Non-Functional requirements. Is the product right for the customer?
System testing is the final set of verification tests after Unit and Integration tests. Acceptance testing is making sure the customer understands how the product solved their problem or the Validation. Using Use Cases helps ensure that the validation process is catching the needs of the customer.
