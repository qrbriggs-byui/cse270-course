Notes for Classroom

Overview

Most testing will fall into different stages of software lifecycles. This week is a follow on of Week 03 & 04. We will explore which types of tests (04) will be completed in which phase of the lifecycle (03).

  • Students must understand that each level of testing effects the next level of testing, and changes in one can cause issues up the chain.
  • Students will explore the responsibilities of a Tester in each phase of the software lifecycle. Responsibilities are not entirely located at the end of the DevOps portion of the lifecycle.

This week we are expanding from Test Case, that deals with Unit level, to the System and Integration level of the Software Development lifecycle phases. The focus is on Regression Tests. There is not enough time to test every possible variable, code path, and way a user could use a complex software system. A Regression Test is a small subset of all the possible tests. They say, "make it idiot proof and there is always a smarter idiot." A regression tests is more than the happy path (options that don’t produce any errors), it also tests performance, robustness, and fault-tolerance. Regression tests are mainly used in Software Maintenance and in the Regression, tests is derived from the Acceptance Tests or Coverage Tests. From the 1st Iteration, a regression plan should be in place to prevent error creeping in to the baseline, this holds true for from Spiral to Test Driven Development to Agile methodologies.

By the end of the week, each student will be able to:

What is regression testing?
Regression testing is the compilation of Unit, Integration, System, and Acceptance testing. As mentioned Regression test is a small subset of all of these tests. The regression test’s purpose is to make sure all the program’s vital elements are working as the project starts to integrate the parts; this includes workflows established by the customer. The regression test changes as the program grow and changes.
The first version of a project’s regression test starts with the first two Unit Tests. As more Unit Tests are developed, the Regression Test is optimized to test the project’s previous and newly developed features.
For large projects, it is impossible to test every variation of the System, so vital or happy path tests are written to verify new functionality does not change the design.
After Unit Tests, Regression tests start to include Integration tests and System (or End to End tests). After delivery, Regression is again updated, this time with some Acceptance tests.

Coding Automated Tests
The attached code is to help you with creating automated tests.
https://byui-cse.github.io/cse270-course/Labs/CSE270M08TeachLabDrMobilSeleniumCode.py
You need to create a new project and python file. Copy the code into the new python file. In the class:LoadBrowser:setup function, change to your preferred brower’s webdriver.
The code is made to simplify your coding experience. For each test you created, you create a python list with dictionary values.

  • “name”:               The first key-value set is the step name. The key is the name, and the value is what you what to call the step.
  • “test_element”:               The values for this are “step,” “results,” or “validate,” many lists the tests you wrote last week.
  • “variable”:           The value is the name of the variable that you would like to store the results of the step. The variable name is used as an input in any other commands. The first variable is “frame” and contains the HTML from the tc-tiddler-frame or story river. Each time a command is run, the results are put into a python dictionary. The variable name is the key and the results as the value, so be careful to identify the variable names uniquely.
  • “command”:       This name of the command that you use to execute the step.
    • Step commands:             
      • attribute:             get an attribute of html tiddlewiki div tag
      • className:         gets the corresponding html (parameter) contents that has that class name, and puts the results in “variable: name”
      • tagName:            get the HTML tag (parameter) contents and puts the results in “variable:name”
      • click_single:        sends a click command to a button
      • click:                      represents a link being clicked
      • send_keys:         puts in parameter value in a text box
    • Results command:          
      • equal:                   will compare the parameter to the input values
      • exists                    will determine if the input value has information.
    • Verify commands:          
      • equal:                   will compare the parameter to the input values
      • exists                    will determine if the input value has information.
    • Debug commands:
      • text:                      puts the input variable contents as a string, removing all of the HTML tags.
  • “parameter”:
    • Each command can take a parameter to compare to the input.
  • “input”:
    • The input corresponds to the “variable”. The first input of every test should query the “frame” variable.
