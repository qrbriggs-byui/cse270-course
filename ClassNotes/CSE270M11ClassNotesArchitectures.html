Welcome to Week 11: Architecture

This week we will talk about testing architecture. Yes, some of you know a little about software architecture, and some know a lot.

Software Architecture:

Most software and system engineers consider the front-end the user interface or client application. The back-end being considered the database. The middle-end is the server and sometimes considered the back-end. Altogether, this is referred to as full-stack development.  With different architectures, the role of the front and back-end changes a bit.
There are several different types of architectures that we will focus on this week, and there are plenty more.

  • Peer-to-Peer: This architecture is usually a direct connection from one device to another, much like a phone call. There is no going through a central server to transfer information but through a set of nodes.
  • Wireless: Much like most networks, but the signal is broadcast, and the node picks up the information and transmits it via a different medium, i.e., wire/cable. Different forms of wireless include Bluetooth, wireless internet, etc.
  • Client-Server: Widely used for web services. A client, the browser, connects to a specific IP-location to a server. The server provides information back from the client’s requests. Examples include beyond the web, email, most SaaS (Cloud applications), and Movie and File Services. The client can be dump as a TV or smart as your laptop or another server. A Client-Server can be event-driven or have a persistent connection.
  • Rule-Base: This service will provide information based on a set of rules, mostly working with business logic and service-oriented architectures. These architectures sometimes seem that they have a mind of their own. The actions are asynchronous and sometimes difficult to debug. The rules kick off processes base on events, thresholds, and performance.
  • Event-Driven: This architecture is similar to Rule-based, but usually located on the client-side via HMI (Human Machine Interface), also known as GUI (Graphical User Interface) or user applications. The application waits for an event from the user. These are also used on the server-side to fulfill requests from clients.
  • Database: This is usually considered the other side of the back-end of the system. Database Management System can be simple as a text file, to RAIDs, or complex as Oracle. In some instances, the complex ones handling complex events like an Operating System.

Once these architectures are working with all the third-party software connected to the project’s software, tracking issues becomes a little more complex. The best practice is to use log files to determine the workflow of your data.


Flow Diagrams

A good system design will have data, process, work, control, and functional flow diagrams.

  • The data flow diagram identifies all the processes, the type of data they accept and produce, and the connection where the data goes afterward. Data flow perspective is a very common and essential diagram.
  • Process diagrams similar to data flow diagrams but usually indicate event processing. The process will be used when conditions are true.
  • Workflow diagrams describe the people’s process of getting a product complete. The workflow then is worked into the business logic. For example, there is the workflow for checking someone in for an appointment in the medical field, which includes making sure that they have a referral, their insurance is good, diagnosing their symptoms, and finding out about their medical history.
  • Control diagrams describe where the decision-making point of the system. Usually, these are bottlenecks in the workflow or major distribution areas.  
  • Functional flow diagrams are detailed descriptions of what each sub-unit of the program is doing to the product. For example, what functions are producing the addition of frosting to an automated cake machine.

Each of these flows allows the engineer to know and look at the system from different angles, perspectives, and views. Many times developing user-profiles and use cases help create scenarios through different views.
The trick is to know which part of the system should be producing data, and what the data looks like at that stage of the system?

Log Files.

Log files are your friend, especially in testing. Most testing is black-box testing, and you don’t have access to the internals of the program or process in the program. The log files give you that insight.
Log files should provide state transitions, data transformation confirmations, and input and output verifications. A state transition is when the process is only allowed to accept certain input at that time. For example, an elevator at the bottom floor will only accept up requests. Or another example is the application setup wizard will only accept file location for installation. Data transformation is the process of transforming the input to the output. The input and output verification log statements show that the input data is good and acceptable and that the output is right and range.
You follow the data through the log files. You follow what process touched the data using each of the processes log files. You verify that the completion of the user’s workflow by checking the sequence of events found in the log files. You can determine the state and control decisions made during the sequence with the log files. And you can determine coverage of the system or what functional parts of the system processed the data as one variation of the log files.   
Suppose you didn’t catch it. The log files verify the flow diagrams.


Logging Systems

Logging Systems have been, for the most part, standardized and built into coding libraries that integrate with most programming languages. Each logging system will have different levels of logging.

  • DEBUG: The finest grain logging level. Usually, a programmer using this logging statement to output primitive values, entrance and exiting of functions, and state change.
  • INFO/STATUS: The INFO statements provide information about the general health of the system.
  • ALERT: The ALERT statements inform the user that there is an action that is required.
  • WARNING: The system is outside the normal happy path process and is trying to recover.
  • ERROR: This is when the system has varied from the happy path, and the system can still recover.
  • CRITICAL: This is when the system has tried a recovery but was unable at that location, and a reset of the states or program had to be completed.

This week prove you will determine the best logging information that you would like to see for each of the use cases.
