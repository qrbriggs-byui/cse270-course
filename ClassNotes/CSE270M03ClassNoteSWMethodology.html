Class Notes M03: Weekly Topic

Notes for Classroom

Overview

Testing is a major phase of any lifecycle. From requirements development to integrating & delivering valuable products, testing is involved in every aspect of any software lifecycle.  

  • This week will explore each stages of different lifecycles, and the roles, responsibility, and products of Software Testing. 
  • This week is an overview of Software Lifecycles, and not a replacement for CS 272, focus on how testing is used in each of the lifecycles and methodologies we study this week. 

Objectives

By the end of the week, each student will be able to:

  • Summarize how Testing fits in to different S/W Development Lifecycles

Announcement

Welcome to Week 3
This week we are working on how testing works with the software lifecycle and the different ways software is developed through methodologies.
Software has one lifecycle. Many different methodologies.
The software lifecycle starts with a concept. Then development of requirements and design. Once the requirements and design are developed, coding and testing. Finally integration and system and acceptance and delivery of the system.
The order of each phases of the lifecycles varies depending on the methodology, but each phase must be done.

Methodologies – Lifecycle Phase:

  • Code and Fix - Concept Dev, Code and Fix, Release
  • Waterfall – Requirements, Design, Implementation, Testing,
  • Spiral – Objectives, Planning, Risk Analysis, Prototyping, Requirement Validation, Design V&V, Simulations/Models/Benchmarks, Detailed Design, Unit test, Integration Test, Acceptance Test, Spirals 1,2,3,4
  • Incremental – Req, Analysis, Design, Implementation, Testing, Deployment, Maintenace
  • V-Model   - CONOP, Req, Design, Implementation, Testing (Unit, Integration, System, Acceptance), Verification, Validation
  • XP – Release Plan, Iteration Plan, Acceptance Test, Unit Test, Pair Programming 
  • Agile – Brainstorm, Design, Develop, QA, Deployment
  • Continuous Integrations and Development – Agile, Integration, Delivery, Deployment, DevOps

Software has a lifecycle, much like a butterfly or moth. Each phase of the development lifecycle can use different methodologies. Each methodology introduced more detail to each lifecycle phase.

  • Concept of Operation
    • Concept
      • Concept is found in Code & Fix, Spiral, and V-Model methodologies. This is one of the first stages of elicitation of requirments from the customer, finding out what they want and how they think it should work.
    • Software Objectivies
      • Software Objectives is the next stage is figuring out the customer's general goals. Even though only spelled out in Spiral, it is generally used in all methodologies.
    • Risk
      • This is a common practice through out all methodologies, but specifically to Spirial. All mature software projects are constenstly identifying risks to mitigate issues and problems.
  • Requirements
    • System Requirements
      • System Requirements are a high level list of requirements that usually identify hardware, operating system, communication, and architecture specifications. Specifically mentioned in Waterfall, System Requirements are needed in all aspects of the Software Development Lifecycle
    • Software Requirements
      • Software Requirements are specification specific to the software being developed. These requirements interact with the system requirements, and identify all the functionality of the the software being developed. Waterfall and Spirial specifically call these out.
    • Architecture
      • The Architecture phase is the part of the System Requirements and is used to beter define what resources the software will run on. Found mostly in V-Model explainations.
    • Requirements Validation
      • Requirements Validation lay out the preliminary research for testing the System and Software Requirements. It is necessary to make sure that the requirements can be tested and that the requirements reflect what the customer wants. Typically this is done through a Software Requirements Review(SRR) meeting, and is a major milestone in the Software Lifecycle. This phase is specifically defined in the Spirial Methodology.
  • Design
    • Analysis
      • Analysis is usually started during the requirements phase and finished during development phase. It takes the requirements and starts the transformation process breaking the requirements down a point where programmers can start implementing it in code. This process also address product, customer, data, and process workflows.
    • Program Design
      • The Program Design is a process of taking the requirements and coming up with solutions on how to implement it in code. Introduced in waterfall, this is found also in Spirial, and V-Model, also known as Product Design and Detailed Design.
    • Prototyping
      • Prototyping is a technique of creating representatives of the software functionality to show to the customer. Spiral methodology heavly relies on the technique, and is used in Demostrations during agile. Prototypes do not typically actually work, and care should be take to prevent customer thinking that it is the current working product.
    • Product Validation and Verification
      • As part of the last step in the Design Phase, culumating in a Software Design Review meeting (Waterfall & Spiral), the team makes sure that the customer wants/requirements are fulfilled by the design.
  • Implementation
    • Coding/Implementation
      • Coding and Implemention is where the tires meet the road. It is the test if the Requirements and Design are detailed enough to translate the customer's needs down to executable binary code. Found in all methodologies it is the most important lifecycle phase.
    • Build
      • The Build phase is more complicated then just compiling the code. It involves making sure that it works with other parts of the System, Architecture, and Design. It properly links all the parts together in to a simplied processes. This phase has recently been identified in the Continious Integration and Delivery process.
  • Testing
    • Unit Testing
      • Unit Tests are typically written (before or) during (or after) the coding and implementation phase. Its main focus is on Black Box testing each of the small coded components to verify that they work correctly.
    • Simulations/Simulators
      • Much like prototyping, simulator and simulations provide a way to verify that the system, architecture and software is working correctly. Specifically spelled out in spiral, simulators help when the hardware is not available, and simulations model the system to provide performance benchmarks, and is a valid testing method -
      • Testing by Analysis.
    • Integration Testing
      • As part of verifying the build phase, Integration Testing makes sure that interfaces between minor and major parts are working together, that software is properly talking to the architecture and the system runs as designed.
    • System Testing
      • Found in Waterfall and V-Model methodologies, System Testing is a set of tests to make sure that the identified workflows are working as expected.
    • Acceptance Testing
      • Spiral and XP methodoloies focus on Acceptance testing as a means to verifing and validating the originals customers needs. Typically just before deployment, a meeting, SAT (Software/System Acceptance Test), is done with the customer where each of the customers needs and concerns are addressed through walkthrought tests and demostrations.
  • Deployment
    • Deployment and Version Release
      • After acceptance with the customer, the software is delivered to the customer. This includes hardware (if needed), and software packages for support, and scripts and programs to configure and setup the software. Concerns of releases, and DevOps are also addressed in this phase. The XP methodology specifically requires a Release plan and Agile addresses Deployment procedures.
