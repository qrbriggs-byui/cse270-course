CSE 270: W02 Lab: Auto-Focus

Now that we understand how the auto-focus system on your camera should work, the final step is to find out how it actually works. To do this, we will create a test plan, conduct our tests, and report on our findings.

Test Plan

A test plan is a series of experiments that we can follow to learn more about the unit under test. We call these experiments "test cases." Each test case is individually described with sufficient detail that anyone can carry out.
A test plan consists of the following elements:  

  • ID - a unique identification of the test plan
  • Plan name: A common name identifier, i.e. Camera Focus Test Plan
  • Description or Summary - notes as to the purpose of the tests
  • Pre-requisites - any previous plans, setup, or resources that are required before starting the test plan
  • Category/Component - focus on which element of the software the test plan focuses on. For example, if a software system has multiple features, this section identifies the specific component or category of quality.
  • Author/POC - who wrote this test plan or who is the Point Of Contact for fixes, questions, or issues with the test plan
  • Requirement/Story mapping set: Identity which requirement, design elements, or story this test is verifying.
  • Tests Cases – Each test plan has one or more test cases. Each test case consists of at least two test steps, Action/Results, and Action/Verify pairs, where Action/Results repeats until you can get to a point where you verify a requirement using Action/Verify.
    • Test Case Name: Unique identification of test case
    • Test Steps- series of tests that verify the software feature
      • Each test contains specific step-by-step instructions, with results of the actions until the feature or requirement is verified.
      • Tests consist of a set of steps: With an (A)ction followed by a (R)esults, until you (V)erify the requirement or feature
    • Action – identifies explicitly what needs to happen in a logical sequence.
      • Results: What were the results of the Action? Success is either pass/fail.
    • …Repeat until…
    • Action
      • Verify: The previous action verified a requirement. – Indicate Pass or Fail

So what makes for a good test case? It should have four properties:

  • Understandable: If a tester other than the author needs to implement or modify a test case, this tester needs to understand the test case. Documentation, unambiguous instructions, and standard use of terminology or procedures help with the understandability of the test case.
  • Valid: Each test case should map directly to a feature or design characteristic of the system. Thus, if an experiment were to fail, it would be easy to see how the failure is tied to a customer or user’s need.
  • Reliable: If a test case has a large number of false-positives (instances when errors are reported when none exist), false-negatives (an error exists that is not reported), or the test case produces weird data due to errors in how it is conducted, then people will trust the test caseless. Ideally, our test case should be repeatable, and the resulting data has little noise.
  • Efficient: A well-written test case should execute with little effort or, better yet, automatically. The longer it takes to complete a test plan, the more maintenance cost.

Much like the PB&J video, make sure your steps are clear, precise and you don't skip any steps.

If you are having trouble coming up with test cases, try the following:

  • What is the minimum focus distance? How far away from the subject must the camera be before the auto-focus work?
  • How does color affect auto-focus?
  • How does movement affect auto-focus?
  • Can the camera focus on something not in the middle of the field of view?
  • Do different textures affect the auto-focus? Consider carpet vs. a wall vs. grass, for example.
  • What happens when you place a filter (screen protector, clear plastic, etc.) in front of the lens?
  • How much light do you need for the auto-focus to work? For example, daylight vs. inside vs. dark.
  • If the camera uses a laser or something similar, what happens when it is covered?
  • If your camera has more than one camera, do they behave differently?

Your test plan should have at least a half-dozen test cases. Make sure each test case includes the precise steps that one must carry out to get the desired data. Feel free to use your or your classmates research to create these test cases.

You test case should verify a requirement, feature or story associated with camera auto-focus. You will need to determine that from your research or your classmates' research. Remember, a requirement use the word "shall", and a story use the format, "As <role>, I want to <requirement/what>, so that <why>".

Ask two individuals to execute your test plan. Both should perform precisely the same actions and give you very similar data.

Conduct the Tests

With the test plan created, the next step is to execute it. For each test case, conduct it as precisely as you can. If you are lax or imprecise about executing each step, you might have reliability issues with your data. In other words, executing the same test case twice might not yield the same results. This is a bad thing!
Note that you may notice errors in your test plan as you conduct a given test. If this is to happen, update your test case with better instructions.
Carefully record your results. Record exactly what you did and what results in you were able to observe. Note that your test case may list the expected outcome of a given action. If the outcome was the same or different from what was expected, record that fact.

Report your Findings

The last part of this lab is to write a report describing your findings. This report consists of each test case in your test plan. There should be six sections in your account if there are six test cases. Each section must have the following information:

  1. Name of the test case
  2. The test case should be listed in your test plan.
  3. Exactly what happened when you conducted the test.
  4. What was learned from the test case? Did the camera behave as you expected it to? Was the collected data within the expected range of values?

Each test case should be between a quarter and a half page.

Submission

You will submit a test plan.

  • Page one will consist of Test Plan ID, Name, Description/Summary, Pre-requisite, Category/Component, Author/POC, Requirement/Story, List of Test Cases
  • Each follow-on page will consist of Test Cases, containing Test case name and case steps (A/R or A/V). You will need a minimum of 6 test cases.
  • Last page: Result Summary: As part of understandable, valid, reliable, and efficient include the results of running your test multiple times.
